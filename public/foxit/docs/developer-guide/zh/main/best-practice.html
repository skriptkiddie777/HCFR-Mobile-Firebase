# 最佳实践

Foxit SDK for Web 是在网络环境的浏览器沙盒中运行。选择正确的网站运维配置及SDK配置方案，可以让Foxit PDF SDK for Web 运行的更快。下面章节将就网站运维优化及SDK配置给出参考性配置建议。

# 运维优化

# 开启压缩算法

网络传输中，为了提高效率，通常使用压缩来减少体积。下表为文件UIExtension.css](../lib/UIExtension.css)及[UIExtension.full.js在各个压缩算法下的体积。

文件 原始大小 gzip压缩 brotli压缩
UIExtension.css 1.2M 213kb 156kb
UIExtension.full.js 2.6M 534kb 443kb

当前，主流浏览器都原生支持gzip压缩算法。部署时开启gzip压缩，可提升网络效率。

谷歌提供brotli压缩算法在压缩率上比gzip有优势。但brotli并非各个浏览器原生支持，要通过js解压配合使用，有时性能损失不及收益，需谨慎使用。

# 开启keepAlive

开启keepAlive作用是减少浏览器与服务器的连接请求，可有效提高资源加载效率。

# 缓存文件

缓存资源文件可减少重复请求的时间，lib](../lib)以及[external中的字体文件建议前端缓存。

# Foxit Web SDK配置

在阅读以下内容之前，你可能想了解为什么要进行配置？为了解释这个问题，我们把PDF归为两类，简单和复杂PDF。 简单PDF，结构清晰易于解析，您不需要做任何配置，直接使用SDK默认的设置就可以快速的渲染，编辑和保存文档。 复杂PDF，比较典型代表是CAD生成的PDF，文档包含了大量的页面对象，注释，字体，以及大尺寸的页面布局。此类文档在解析和渲染过程中，会消耗大量内存以及CPU，同时也影响了文档渲染和修改后保存速度。通过配置，可以提高交互性，便于用户使用。

# 静默模式

静默模式屏蔽了PDF中可交互元素，从而提高文档的渲染效率。当前Web sdk通过判断文档是否只读权限(只读权限：任何修改文档的操作不会被执行，非修改的操作可以成功，如link 注释的跳转action)来开启静默模式，代码样例如下：

<script src="path/to/UIExtension.full.js"></script>
<script src="path/to/allInOne.js"></script>
<script>
    var pdfui = new UIExtension.PDFUI({
        ...
        viewerOptions:{
            customs: {
                getDocPermissions: function () {
                    return 0;// 0表示只读，请参阅api手册User_Permissions对象
                }
            }
        ...
    })
</script>

或者

<script src="path/to/PDFViewCtrl.full.js"></script>
<script>
    var pdfviewer = new PDFViewCtrl.PDFViewer({
        ...
        customs: {
                getDocPermissions: function () {
                    return 0;// 0表示只读，请参阅api手册User_Permissions对象
                }
            }
        ...
    })
</script>

# brotli压缩

web sdk的核心是emscripten编译的wasm/asm模块。wasm/asm模块文件体积为8M/13M，加载时长因浏览器性能而异。受限于网络带宽，传统上web应用有压缩文件以提高网络传输的习惯。WEB SDK也有对应配置：加载核心模块使用的brotli压缩。值得注意的是：brotli算法并非浏览器原生支持，使用js代码解压存在一定性能损耗。如果web sdk部署环境带宽比较大，建议用户取消使用brotli以提高更新后第一次加载的速度。使用方式：

<script src="path/to/UIExtension.full.js"></script>
<script src="path/to/allInOne.js"></script>
<script>
    var pdfui = new UIExtension.PDFUI({
        ...
        viewerOptions:{
            jr: {
                brotli:{
                    core:false,// 默认值true，表示开启压缩，false表示不压缩
                }
            }
        ...
    })
</script>

或者

<script src="path/to/PDFViewCtrl.full.js"></script>
<script>
    var pdfviewer = new PDFViewCtrl.PDFViewer({
        ...
        jr: {
            brotli:{
                core:false,// 默认值true，表示开启压缩，false表示不压缩
            }
        }
        ...
    })
</script>

# 并行加载worker

从7.1.1版本开始，Foxit PDF SDK for web在 "lib" 目录下提供了一个脚本文件 "preload-jr-worker.js"，该脚本文件是web SDK在后台执行任务的worker线程。它提前加载JR引擎资源，获取潜在加载性能优势，节省初始化的时候。并且它可以与SDK JS库同时运行，在不同的线程中初始化数据。在HTML index文件中设置 "preload-jr-worker.js" 的主要代码示例如下所示：

<body>
    <div id="pdf-ui"></div>
    <script>
        var licenseSN = "Your license SN";
        var licenseKey = "Your license Key";
    </script>

    <!-- Add the preload-jr-worker.js-->
    <script src="./lib/preload-jr-worker.js"></script>
    <script>
        var readyWorker = preloadJrWorker({
            workerPath: './lib/',
            enginePath: './lib/jr-engine/gsdk',
            fontPath: './external/brotli',
            licenseSN: licenseSN,
            licenseKey: licenseKey
        })
    </script>

    <script src="./lib/UIExtension.full.js"></script>
    <script>

        var pdfui = new UIExtension.PDFUI({
            viewerOptions: {
                libPath: './lib', // the library path of web sdk.
                jr: {
                    readyWorker: readyWorker,
                }
            },
            renderTo: '#pdf-ui', // the div (id="pdf-ui").
            appearance: UIExtension.appearances.adaptive,
            addons: [
                '```'            
            ]
        });
        ...

# Tiling Size

渲染页面时，会进行光栅扫描，如果页面幅度过大，会造成性能消耗。综合情况下，tiling size为1200px时，表现最优。使用方式：

<script src="path/to/UIExtension.full.js"></script>
<script src="path/to/allInOne.js"></script>
<script>
    var pdfui = new UIExtension.PDFUI({
        ...
        viewerOptions:{
            tileSize:1200,
            ...
        }
        ...
    })
</script>

或者

<script src="path/to/PDFViewCtrl.full.js"></script>
<script>
    var pdfviewer = new PDFViewCtrl.PDFViewer({
        ...
        tileSize:1200,
        ...
    })
</script>

# Tiling Size与缩放大小

当前web sdk打开文档的的默认大小为fitWidth，对于页面尺寸大而相对简单的文档，fitWidth可提高渲染速度。然而，fitWidth存在一个弊端，如果文档页面中的对象过多，那么相同高度的可视范围内需要渲染的页面对象也多。通过放大比率和tile size大小的设置可以可减少渲染对象，提高渲染速度。 以可视范围800600，页面对象30004000，tiling大小为200为例：

  • fitWidth下，页面左上角为原点，3000*2400范围内的页面对象均被渲染
  • 0.5倍比率，那么页面左上角为原点，仅1600*1200范围内的对象被渲染
  • 1倍比率，那么页面左上角为原点，仅800*600范围内的对象被渲染 使用方式如下：
<script src="path/to/PDFViewCtrl.full.js"></script>
<script>
    var pdfviewer = new PDFViewCtrl.PDFViewer({
        ...
        defaultScale = '0.5',// or '1'
        tileSize:200,
        ...
    })
</script>

# 渲染模式

PDF文档有高保真的特点，但高保真对性能要求比较高。为了平衡速度与高保真，web sdk对部分注释及表单提供了两种渲染模式：native及canvas。

  • native表示使用核心库gsdk渲染，由于涉及到页面对象的渲染，性能要求比较高。
  • canvas表示使用html canvas引擎渲染，存在一定失真，但速度快，交互性强。

用户如果对注释及表单的高保真要求较低，而速度要求较高则开使用canvas模式。

支持以上两种模式的注释及表单包括：

  1. 高亮
  2. Note
  3. FileAttachment
  4. Sound
  5. PushButton
  6. RadioButton及CheckBox
  7. TextField
  8. comboBox

配置样例：

<script src="path/to/PDFViewCtrl.full.js"></script>
<script>
    var pdfviewer = new PDFViewCtrl.PDFViewer({
        ...
        annotRenderingMode:{
            highlight: 'canvas', // 快速模式
            choiceButton:'canvas',// 快速模式
            pushButton:'native',//精确模式
            textField:'native',//精确模式
            comboBox:'canvas',// 快速模式
            sound:'canvas',// 快速模式
            fileAttachment:'canvas',// 快速模式
            note:'canvas',// 快速模式
        }
        ...
    })
</script>

注意：

  • 在SR下由于使用native渲染需要实时请求服务器，内部默认使用了canvas（不可配置）。
  • 部分文档的text表单使用了内嵌字体显示，使用canvas模式无法精确显示，此种情况建议使用native
  • 部分文档的Push Button表单使用了图标，用于显示二维码等，使用canvas模式无法精确显示,此种情况建议使用native

# 文档加载方式

在了解文档加载方式前，简单介绍一下websdk内存管理。普通js的内存开辟与释放由浏览器管理。当发生内存事件（开辟/释放）时，浏览器会进行内存扫描，释放不再使用的内存。而c++编译出来的wasm/asm对内存操作过于频繁，为了减少内存扰动，wasm/asm使用了一个比较大的arrayBuffer来模拟内存，wasm/asm模块的大部分内存存在于此arraryBuffer中。因此，在加载文档的过程中，需要从普通js内存加载数据到wasm/asm内存。这里有两个因素需要考虑：1、频繁从普通js内存加载数据到wasm/asm内存速度稍慢；2、wasm/asm内存只增不降。鉴于以上，web sdk 提供了三种文档加载方式。

  1. 同步方式

先获取文件的完整二进制流用于加载。内存与性能折中的加载方式，对于50M~300M的文档 建议此种方式。

<script src="path/to/UIExtension.full.js"></script>
<script src="path/to/allInOne.js"></script>
<script>
    var pdfui = new UIExtension.PDFUI({...})
    var blob = getBlob();
    pdfui.openPDFByFile(blob)
</script>
  1. 按需加载方式

无需完整的文件流，仅在加载时获取对应的部分。当文件过大，无法全部放于内存时，或者仅使用文档部分内容时，使用此方式以获得好的体验。由于读取速度较慢，此方式效率稍低。

<script src="path/to/UIExtension.full.js"></script>
<script>
    var pdfui = new UIExtension.PDFUI({...})
    pdfui.openPDFByHttpRangeRequest({
        range:{
            url:'../../../docs/FoxitPDFSDKforWeb_DemoGuide.pdf',
        }
    })
</script>
  1. 内存方式

将整个文件流存储于wasm/asm内存中加载。对于位于本地的小文档，或者可短时间内获取到整个文档流时建议此方式加载。具有读取效率高，加载速度快的优势。在构造PDFUI时，传入回调函数getLoadingMode(),当返回1时表示使用内存方式加载。

<script src="path/to/UIExtension.full.js"></script>
<script>
    var pdfui = new UIExtension.PDFUI({
        ...
        customs:{
            getLoadingMode:function(file){return 1}
        }
        ...
    })
</script>

如果用户实现了自己的file open control, 使用如下方式：

var pdfui = new UIExtension.PDFUI({...})
...//event bind context
{
    var arrayBuffer=getArrayBuffer();
    pdfui.openPDFByFile(arrayBuffer);
}
...
